The Jakarta Faces TCK runs as part of every Mojarra release. During the Faces 4.0 era the full run on Jenkins CI took well over 3 hours, nearly 3 and half hours. Today, on the same hardware, it runs in well under half a hour. This article walks through how we got there.

The starting point: Ant-driven sequential TCK

The original Faces TCK was inherited from the JSF 1.x days. It ran via an Ant harness that deployed each test as its own WAR, started GlassFish, ran the test class against the deployed WAR, undeployed, and moved on to the next. With around 1500 test classes and roughly 8 seconds of deploy/undeploy overhead per test, the bulk of those 3 hours was not test execution but Ant orchestration and GlassFish redeploys.

The harness was also opaque to Maven; it ran as a black-box external process. You couldn't mvn -pl :failingTest your way to a single test, you couldn't easily attach a debugger, and you couldn't parallelize without rewriting the harness.

Migration to the new selenium-based TCK

Pull requests #2146, #2147, #2148, #2149 and #2150 migrated the entire old TCK, in five batches, to a new structure: every test is a JUnit 5 + Arquillian + Selenium integration test, every group of related tests is a Maven module producing a single WAR, and Arquillian deploys that single WAR once per module instead of once per test.

The migration was almost entirely mechanical. Test logic was with help of Claude Code rewritten from HtmlUnit assertions to Selenium WebElement assertions, with existing helpers like guardHttp(Runnable) and guardAjax(Runnable) wrapping the actions that trigger a non-ajax page load or an ajax response respectively. The xhtml pages were copied verbatim where possible, since the rendered output is what the spec mandates and changing it would change what's tested.

The bulk of the time savings here came not from the assertion technology but from WAR consolidation. Where the old TCK had one WAR per test, the new TCK has one WAR per logical feature group (composite components, ajax behaviors, flows, etc). For a feature group containing 30 tests, that's 30 deploys collapsed into 1.

Per-version test pruning

Pull requests #2152 and #2153 added auto-skip support for CSP-related tests on Mojarra 4.0.17+ and 4.1.8+. The CSP feature was backported from Faces 5.0 into the 4.x branches, but a few specific tests required behaviour that wasn't part of the backport. Rather than skipping the entire CSP suite on 4.x, the autoskip looks at the running Mojarra version and disables only the affected tests. Mojarra 5.x runs them all.

gf-pool: a shared GlassFish pool with slot leasing

Pull request #2156 introduced the biggest architectural change. The new TCK still deploys one WAR per module, and previously each module's failsafe-forked JVM started its own GlassFish, ran the tests, and stopped GlassFish. With 100+ modules, that's 100+ GlassFish JVM starts, each costing several seconds.

The gf-pool module pre-starts a configurable pool of GlassFish instances, all on different port ranges, in gf-pool/target/pool/slot-N/. A small Java agent (SlotLeaserAgent) is attached to each failsafe-forked test JVM. The agent acquires an exclusive FileLock on slot-N/lock, publishes that slot's port numbers as system properties for the Arquillian remote container, and holds the lock for the JVM's lifetime. When the JVM exits, the lock is released and the slot becomes available to the next module.

The pool grows on demand up to -Tn slots, so a sequential build uses one GlassFish; mvn -T4 install grows to four. A custom Ant task installs a JVM shutdown hook on Maven's own JVM, so all slots get stopped at session end (or on Ctrl+C), no orphaned processes.

The result on Jenkins CI was a drop from 3h 19m to 1h 18m at -T4. The full reactor still runs every test exactly once; it just reuses the GlassFish JVMs across modules instead of throwing them away.

Framework cleanup: implicit waits and dead CDP sessions

Pull request #2157 attacks two long-tail Selenium pathologies that the gf-pool work exposed.

The first is Selenium's implicit wait. The TCK had driver.manage().timeouts().implicitlyWait(STD_TIMEOUT) with STD_TIMEOUT = 10s, on top of an explicit WebDriverWait. Mixing the two is documented as unreliable, and on findElements() it has a sharp cost: when the result list is legitimately empty (no matching child element), Selenium polls the DOM for the full 10 seconds before returning the empty list. A handful of TCK tests that walk an element's children and assert "no element children, just text" therefore paid 10 seconds per assertion.

The most extreme example was Spec1553IT, a single test asserting that 24 different XML namespace declarations all render correctly. Each assertion called element.findElements(By.xpath(".//*")).size() == 0 on a div whose only "child" was the rendered text "value". 18 such assertions, each taking exactly 10 seconds, made a single test method take 207 seconds. Setting implicit wait to Duration.ZERO and dropping the findElements wrap brought it to 12 seconds. findElement() still polls via the explicit wait, so "wait for an element to appear" semantics are preserved.

The second pathology is the dead-CDP-session retry storm. Selenium's DevTools.createSession() occasionally times out under heavy parallel load on Jenkins. The original code swallowed that timeout and proceeded to call Network.enable() on the now-dead session, with a 10-attempt exponential backoff. Each attempt waited the ~30 second sendAndWait timeout plus the backoff sleep, totalling around 491 seconds before giving up. We saw 5 separate test classes hang for exactly 491 seconds in the same Jenkins run. The fix propagates the createSession() failure so DriverPool can quit() the broken driver and create a fresh one; cost drops to ~32 seconds.

Module splits: parallelism over deployment count

Same PR, two big module splits. faces20/api/api-main was a single WAR with 132 IT classes; that's a lot of work for one parallel slot. Split into 5 sibling modules grouped by top-level package (component, event, converter-validator, misc, app), the slowest piece dropped from 3:50 to 2:53. Each child module has its own WEB-INF; shared resources go into a small common-tckjar served via the Servlet 3+ META-INF/resources overlay so all WARs see them without duplication.

Same trick on faces20/renderkit: one WAR with 27 component renderer tests became renderkit-input, renderkit-output, renderkit-structure plus a shared common-tckjar for the test beans. From 5:30 to 1:00 for the slowest child.

You'd think more modules means more deploys means more cost. The trade-off is real but small: each new module adds roughly 30 seconds of deploy overhead, but those deploys run in parallel on the gf-pool slots. As long as the total slot demand stays below the pool cap, splitting only buys parallelism.

Negative-path ajax assertions

Some tests deliberately verify that an action does not trigger an ajax response. The old idiom was:

try {
    page.guardAjax(() -> input.sendKeys("1", Keys.TAB));
    fail("There should be no ajax behavior");
}
catch (TimeoutException e) {
    assertEquals("", output.getText());
}

This costs STD_TIMEOUT (10 seconds) per call to prove no ajax fired. Issue2439IT did this three times in a single test method and elapsed 32 seconds.

The new helper WebPage.assertNoAjax(Runnable) registers a faces.ajax.addOnEvent listener that flips a window flag when any ajax callback fires, runs the action, sleeps 500ms, and returns whether the flag stayed unset. Issue2439IT dropped to 9 seconds.

Polling helper removal

The TCK had several "is text in the page" helpers (isInPage, isInPageText, isNotInPage, etc.) that wrapped a contains check in a WebDriverWait. These were leftovers from the HtmlUnit-based predecessor of the new TCK, before guardHttp and guardAjax existed. For negative assertions (isNotInPage) the timeout always fires; you pay 10 seconds per call to prove the text is absent, even though the preceding guardHttp already made the page settle.

Three of the helpers (isNotInPage, isNotInPageText, isInPageTextReduced) were removed and their 6 call sites migrated to direct (page.getPageText() + page.getInputValues()).contains(text) checks. The two remaining ones (isInPage, isInPageText) have 130+ call sites between them; they're queued for a follow-up.

api-component split: hoisting Html*TestServlet up into a shared tckjar

Pull request #2158 resolved the 2:53 floor. The hoist was straightforward: UiInputTestServlet and friends moved from the api-component WAR into api-component-tckjar, and the original WAR became a parent pom aggregating four sibling WARs grouped by feature area (component-html-input, component-html-output, component-misc, component-ui). Wall on the long child dropped from 2:53 to under 1:30.

Same PR consolidated tiny modules where the per-module overhead exceeded the test work (faces40 from fourteen modules into five, faces41 from four into one), split faces22/Ajax into ajax-inputs and ajax-lifecycle, and split faces20/renderkit further into renderkit-input, renderkit-output, renderkit-structure. Local -T8 went from 4:21 to 3:58; -T4 from 6:38 to 5:58.

HttpClient connection reuse

The Selenium BaseITNG.getResponseBody(String) helper was instantiating a fresh java.net.http.HttpClient on every call. Each HttpClient.newHttpClient() spins up a selector thread and a worker executor; the cost is barely visible on Linux where socket setup is cheap, but on macOS where Mach networking has slower TCP setup it was burning a noticeable share of the api-application and api-component-* wall. A one-line change to cache the client as a static field cut a M1 Pro -T5 run from 5:58 to 4:47.

Per-test progress visibility

Maven's per-module test output at -Tn is buffered until the module finishes, which is fine for a green build but unhelpful when something hangs. ProgressListener is a JUnit Platform TestExecutionListener registered via the META-INF/services SPI. It prints one colour-tagged line per test event:

[RUNNING][SLOT1][faces20-api-application-misc][1/35] SecretIT#secretRenderPassthroughTest()
[SKIPPED][SLOT3][faces22-ajax-inputs][4/95] Issue3833IT#test  reason: ignored at the request by myfaces
[FAILED][SLOT3][faces20-api-component-html-input][5/123] HtmlCommandButtonIT#actionTest  exception: ...

The slot tag identifies which gf-pool slot the failsafe-forked JVM is leasing, read from the gf.pool.slot system property the agent publishes. The module tag reads ${project.build.finalName} with the test- prefix stripped. The progress counter reports the per-JVM count out of the total test plan, which with reuseForks=false maps one to one to "test N of M within this IT class". Both tags degrade gracefully to absent if their underlying property is missing, so the listener works unchanged when vendors run the TCK against their own server outside the gf-pool environment. On failure the full stack trace is printed inline, so the build log is sufficient for triage even when target/failsafe-reports is incomplete from an aborted run.

The numbers

  • Faces 4.0 era, sequential Ant-driven TCK, Jenkins: 3h 19m; local: 2h 57m
  • After #2146-#2150 (selenium migration), Jenkins: 1h 18m; local: 1h 5m
  • After #2156 (gf-pool), Jenkins at -T4: 24:23; local at -T4: 10:13; local at -T8: 6:49
  • After #2157 (framework cleanup + first wave of module splits), local at -T4: 10:13 → 6:38; local at -T8: 6:49 → 4:21
  • After #2158 (api-component split + further consolidation), local at -T4: 6:38 → 5:58; -T8: 4:21 → 3:58

Wall is now bottlenecked by the natural minimum: pool start, the longest test module, pool stop. Further wall improvements need either a faster individual test (the isInPage/isInPageText migration noted above is one such), more CPU headroom on the build host, or convincing the Eclipse Foundation to raise the cpu=2 quota on the Mojarra release pipeline's Jenkins agent.

Info: Jenkins runs Kubernetes, "2CPU", 16GB; local machine runs i9-10900x, 32GB DDR4, U.2 SSD.
